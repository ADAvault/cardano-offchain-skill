# API Resilience and Failover

This document covers the client-side API resilience pattern used in production:
automatic failover between primary and DR endpoints with timeout-based switching,
CORS proxy architecture, and the data pipeline that feeds the frontend.

## API Failover Architecture

The frontend makes all data requests through a resilient fetch wrapper that handles
timeouts and automatic failover:

```
Browser (Cloudflare Pages static site)
  |
  v  client-side fetch
api.example.com --> Cloudflare --> reverse-proxy (nginx) --> api-server:3001 (Express)
  |
  v  failover (5s timeout)
api2.example.com --> Cloudflare --> dr-proxy (nginx) --> dr-api-server:3001 (Express)
```

### Primary/Fallback URL Configuration

```typescript
const isDev = import.meta.env.DEV;  // Astro/Vite environment variable

const API_PRIMARY = isDev
  ? 'https://api-dev.adavault.com/api/v1'   // Dev API with localhost CORS
  : 'https://api.adavault.com/api/v1';

const API_FALLBACK = 'https://api2.adavault.com/api/v1';
```

In dev mode, the primary points to a dev-specific API that allows `localhost` CORS
origins. The fallback is always the DR API at `api2`.

### The `apiFetch()` Function

```typescript
export async function apiFetch(path: string, options?: RequestInit): Promise<Response> {
  const normalizedPath = path.startsWith('/') ? path : `/${path}`;
  const primaryUrl = `${API_PRIMARY}${normalizedPath}`;
  const fallbackUrl = `${API_FALLBACK}${normalizedPath}`;

  try {
    const controller = new AbortController();
    const timeoutId = setTimeout(() => controller.abort(), 5000);

    const res = await fetch(primaryUrl, {
      ...options,
      signal: controller.signal,
    });

    clearTimeout(timeoutId);

    if (res.ok) return res;
    throw new Error(`Primary API error: ${res.status}`);
  } catch (err) {
    // In dev mode, don't try fallback (api2 won't be available locally)
    if (isDev) throw err;

    console.warn('Primary API failed, trying fallback:', err);
    return fetch(fallbackUrl, options);
  }
}
```

### Design Decisions

**5-second timeout with `AbortController`**: If the primary API doesn't respond within
5 seconds, the request is aborted and the fallback is tried. This covers network
issues, server hangs, and Cloudflare 52x errors.

**4xx responses do not trigger failover**: The current implementation throws on any
non-ok response. A refined version should return 4xx responses to the caller since
they represent client errors (bad request, not found) that the fallback would also
reject:

```typescript
// Improved: only failover on 5xx or network errors
if (res.ok || res.status < 500) return res;
throw new Error(`Primary error: ${res.status}`);
```

**Dev mode skips fallback**: The DR API (`api2`) has its own CORS whitelist that
doesn't include `localhost`. Attempting fallback during development would fail with
a CORS error that obscures the real problem.

**No timeout on fallback**: The current implementation doesn't timeout the fallback
request. If both APIs are down, the user waits for the browser's default timeout
(typically 30-60s). A production enhancement would add a timeout to the fallback too:

```typescript
// Enhanced: timeout on fallback too
const ctrl2 = new AbortController();
const tid2 = setTimeout(() => ctrl2.abort(), 5000);
try {
  return await fetch(fallbackUrl, { ...options, signal: ctrl2.signal });
} finally {
  clearTimeout(tid2);
}
```

## Koios CORS Bug

The Koios public API (https://api.koios.rest) does not send proper CORS headers,
causing browsers to reject direct requests. This is a known issue:
[koios-artifacts #397](https://github.com/cardano-community/koios-artifacts/issues/397).

**Solution**: Proxy Koios requests through your own API:

```
Browser --> api.adavault.com/api/v1/account/:stakeAddr/delegation
                |
                v (server-side, no CORS)
         api.koios.rest/api/v1/account_info
```

The Express API acts as a CORS-safe proxy, adding proper `Access-Control-Allow-Origin`
headers.

### CORS Ownership: Express, Not nginx

A critical architectural decision: Express owns all CORS headers. The nginx reverse
proxy does NOT add CORS headers. If both Express and nginx add CORS headers, browsers
see duplicate `Access-Control-Allow-Origin` headers and reject the response.

```
# WRONG: Dual CORS headers
nginx:   Access-Control-Allow-Origin: *
Express: Access-Control-Allow-Origin: https://adavault.com
# Result: Browser rejects (multiple values)

# RIGHT: Express only
nginx:   (no CORS headers -- pass through)
Express: Access-Control-Allow-Origin: https://adavault.com
# Result: Browser accepts
```

## Data Pipeline

The API serves data generated by an on-prem cron pipeline:

```
batch-scripts (cron on build server)
  |
  v  SSH/SCP to web servers
JSON files on ~/api-server, ~/dr-api-server
  |
  v  Express reads files (60-second cache)
API responses
```

### Data Files

| File | Content | Frequency |
|------|---------|-----------|
| `ALL-epochStats.json` | Pool stats, saturation, delegators | Every 10 min |
| `ALL-rewardTotals.json` | Lifetime and recent APY per pool | Daily |
| `ALL-predictedPerf.json` | Block performance predictions | Every 10 min |
| `{pool}-leader-summary.json` | Block schedule per pool | Every 10 min |
| `{pool}-poolhistory.json` | Last 20 epochs rewards/blocks | Daily |
| `ADA-USD-price.json` | ADA/USD price feed | Every 10 min |
| `ADA-BTC-price.json` | ADA/BTC price feed | Every 10 min |

## Frontend Usage

The `apiFetch()` function is used throughout the frontend for all API calls:

```typescript
import { apiFetch } from '../config';

// Fetch pool data
const res = await apiFetch('/pools');
const data = await res.json();

// Fetch delegation status (proxied Koios call)
const res = await apiFetch(`/account/${stakeAddress}/delegation`);
```

The wallet integration hook has its own copy of the failover logic (independent of
the shared `config.ts`) to keep the wallet module self-contained. Both use the same
pattern: 5s timeout on primary, automatic fallback to api2.

## Rate Limiting

nginx on the reverse proxy enforces rate limits:

| Endpoint | Limit | Burst |
|----------|-------|-------|
| API (`api.adavault.com`) | 10 req/s | burst 20 |
| Data (`data.adavault.com`) | 30 req/s | burst 50 |

## Incident Response

For API outages, check in order:

1. Health endpoint (`curl https://api.adavault.com/health`)
2. PM2 status on API server
3. Local API connectivity (`curl http://localhost:3001/health`)
4. nginx reverse proxy status and error logs
5. Proxy-to-API connectivity
6. SSL certificate expiry
7. Firewall rules (UFW on API server)
8. nginx access log gaps (no requests = upstream network issue)

A 523 error with no nginx errors means traffic never reached the origin server --
it's a network issue upstream of your infrastructure (Cloudflare or ISP).
